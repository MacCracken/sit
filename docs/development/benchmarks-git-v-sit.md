# Benchmarks: git vs sit

Head-to-head comparison on a real workstation. sit is a first-party Cyrius implementation — every layer (SHA-256, zlib, object store, signing, wire protocol) is written from scratch with no FFI to C libraries. The point of this document is to be honest about where sit currently wins, loses, and breaks even against git, and to track that over time as sigil/sankoch/patra mature.

## Setup

- **Host**: Linux 6.18.22-1-lts x86_64
- **git**: 2.53.0 (system install, libpcre2 + libz-ng + libc dynamic deps)
- **sit**: 0.6.0 built from source at `src/main.cyr` via `cyrius build`
- **Timing**: `date +%s%N` around the command (~nanosecond resolution). Each operation runs with a fresh scratch repo and is measured 10–15 times; we report **min** (closest to true operation cost, minus noise) and **median** (typical case) in milliseconds.

Reproduce with:

```sh
cyrius build src/main.cyr build/sit
SIT=$PWD/build/sit ./scripts/benchmark.sh
```

## Binary footprint

| | size | dynamic deps |
|---|---:|---|
| `git` (primary dispatch binary) | **4,523,048 bytes** (~4.4 MB) | libpcre2, libz-ng, libc |
| `/usr/lib/git-core/*` (all 183 sub-binaries) | **7,390,732 bytes** (~7.4 MB) | same |
| `build/sit` (one statically-linked binary) | **~608 KB** | **none** |

**sit ships one ~608 KB statically-linked binary with zero dynamic dependencies.** The primary git binary is ~7× larger; the full git install footprint is ~12× larger. sit bundles SHA-256, zlib-compatible compression, ed25519 signing, its own object store, and its own wire protocol directly into the executable — no libpcre, no libz, no dispatch subcommand binaries.

- **`status` on 100 files / 100 commits** and **`clone` of 100-commit repo** — `sit status` SIGSEGVs on repos at this scale (tracked as audit finding **S-33**, URGENT for v0.6.1). The clone bench hits the same crash via `materialize_target`. Both rows come back as soon as S-33 is fixed.

## What the numbers mean

**sit wins on `init`, `commit`, and `fetch`.** Small-op latency benefits from sit's static build (no dynamic linker startup, no fork+exec dispatch through `git-core`). `fetch` in particular is 5× faster on the local-path transport because sit copies raw compressed bytes directly DB-to-DB via patra, while git walks pack indices through pipe-based subprocesses.

**`add` and `diff` regressed against git between v0.5.1 and v0.6.0.** v0.6.0 added input validators (tree-entry-name, refname, hex-prefix) and size caps on every call into the object store. The `add` regression on large blobs still lands squarely on sigil's software SHA-256 throughput (benched in isolation at ~12 MB/s in `tests/sit.bcyr`). Once sigil grows a SHA-NI or equivalent fast path, this gap collapses.

**`log` regressed** because v0.6.0 now sanitizes author-identity bytes (S-21) and gates every ref-file read through `hex_prefix_valid` (S-11). The cost is per-commit; the remedy is on the roadmap as the patra-DB-handle-caching refactor (P-01), which will open the object DB once per `sit log` instead of once per commit.

## Honest caveats

- **One machine, one run.** These numbers are a snapshot, not a study. Re-run on your own hardware before quoting. The bench script is at `scripts/benchmark.sh`.
- **sit has a known `status` crash at scale.** Caught by this benchmark run and logged as S-33. Small repos are fine; the specific `sit status` path that walks a multi-commit + multi-file tree is the affected codepath. v0.6.1 fix.
- **sit is young.** The object store isn't optimized yet — the single biggest perf improvement available (audit finding P-01) is caching the patra DB handle process-wide instead of opening it per `read_object`. Work queued for v0.6.x.
- **git is doing more.** The git install bundles pcregrep, rebase, gc, merge-base, diff-tree, hundreds of plumbing commands. sit has 24 so far — the size comparison is not "git does the same work in 12× the bytes." It's "sit covers the core VCS loop in 12× less disk."
- **sit's network transport is local-path only.** v0.6.0 fetch-1commit number is flattering because both git and sit are going through the filesystem. When sit grows HTTP / SSH (v0.7.0) the network number will be a new comparison axis.

## Methodology

1. For each operation, build a **fresh** scratch repo every run — no warm-cache advantages either way.
2. Run the operation N times (10 for heavy ops, 15 for light ones). Record each wall-clock time with `date +%s%N`.
3. Report **min** (approaches the true CPU cost, filtering out scheduler noise) and **median** (typical case).
4. Author identity is pre-configured on both sides (`Bench <b@e>`); no credential fetching in the hot path.
5. Benchmarks run serially on a quiet workstation; no background load beyond the normal desktop session.
6. `RUNS_LIGHT` and `RUNS_HEAVY` environment variables override the default iteration counts.
