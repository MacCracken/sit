# 2026-04-24 — P(-1) Security + Performance Audit

**Target**: sit 0.5.1 (post-refactor, 13 files under `src/`, 24 commands)
**Scope**: Internal review (every `src/*.cyr`) + external CVE mapping (git, Mercurial, zlib, Jujutsu) + performance / memory review.
**Methodology**: Three parallel agents (security internal, perf/memory, CVE external) plus a targeted manual re-read of `cmd_clone`, `parse_tree`, `materialize_target`, `resolve_hash`, `config_file_set`, `read_remote_ref`. Findings cross-checked between sources before inclusion.

---

## TL;DR

sit has **4 CRITICAL** and **11 HIGH** severity issues concentrated in three attack surfaces — the wire protocol's local clone (v0.5.0 new), tree-entry handling during `materialize_target`, and unsanitized input flowing into ref-path concatenation and config writes. Fix these before any network transport (HTTP/SSH in v0.6.x) lands — HTTP exposes the same primitives to remote attackers, not just local ones.

The performance picture is a single architectural wart (per-call patra open) that cascades into ~5 of the top-10 hot paths. One fix (cache the DB handle) unblocks them all.

### Top priorities

1. **S-03** — Tree-entry name path traversal on clone / checkout (CVE-2018-11235 shape) → arbitrary file write
2. **S-01** — SQL interpolation of attacker-controlled hash prefix into `LIKE '<prefix>%'`
3. **S-02** — Ref names written as filesystem paths without validation (affects `branch`, `tag`, `checkout -b`, `remote_add`, and the fetch-receiving side)
4. **S-04** — Wire protocol opens any attacker-chosen path as a patra DB (symlink / arbitrary device node)
5. **S-05** — Config value newline injection (CVE-2023-29007 / CVE-2025-48384 shape) — RCE primitive once any exec-shaped config key lands
6. **P-01** — Every `read_object` reopens the patra DB; cascades to fsck, log, fetch, push, clone, materialize

## Security findings

Numbering: `S-NN`. File paths are repo-relative. Severity scale: **CRITICAL** (RCE / arbitrary write from attacker-controlled input with plausible trigger), **HIGH** (requires modest effort or specific conditions), **MEDIUM** (defense-in-depth, incorrect behavior on malformed input), **LOW** (hygiene).

### CRITICAL

#### S-01 — SQL interpolation of attacker-controlled hash prefix
**File**: `src/object_db.cyr` — `resolve_hash`
`prefix` (user argv in every ref-resolution command) is `memcpy`'d directly into a `LIKE '<prefix>%'` SQL string with no escaping. A prefix containing `'` breaks the query; `'; DROP TABLE objects; --` or similar can mutate DB state depending on patra's SQL robustness. Reachable via `sit show '<payload>'`, `sit cat-file '<payload>'`, `sit log '<payload>'`, `sit verify-commit '<payload>'`, and via malicious remote refs (see S-10 + S-16).
**Impact**: Object store tampering, disclosure, DoS. At minimum, crashes on any `'` in a prefix.
**Fix**: Reject any non-hex char in `prefix` before interpolation. Pre-validate with a `hex_prefix_valid(s)` helper called at the top of `resolve_hash`. Independently, push this up to patra's roadmap as "bound parameters for STR columns."

#### S-02 — Ref names written as filesystem paths without validation
**File**: `src/refs.cyr` (cmd_tag, cmd_branch, cmd_checkout -b), `src/wire.cyr` (write_remote_ref, write_remote_tracking)
Ref names go straight into `.sit/refs/heads/<name>` / `.sit/refs/tags/<name>` / `.sit/refs/remotes/<remote>/<branch>`. No rejection of `..`, leading `/`, embedded `\n`, trailing `.lock`, control chars, or NTFS-reserved sequences. `sit branch ../../../../tmp/evil` writes a ref file outside `.sit/`. `sit push origin ../../evil` writes to the remote's arbitrary path.
**Impact**: Path traversal; write arbitrary files the sit user can write. On the receive side (malicious remote advertises a ref named `../../../etc/cron.d/x`), `write_remote_tracking` writes to that path on our filesystem during `sit fetch`.
**Fix**: Implement `refname_valid(name)` enforcing the full [`git check-ref-format` ruleset](https://git-scm.com/docs/git-check-ref-format): no `..`, no leading `.`, no `.lock` suffix, no control chars, no `~ ^ : ? * [`, no `@{`, no `\`, reasonable length cap. Call at every write site. Document the grammar in an ADR.

#### S-03 — Tree-entry name path traversal on `sit clone` / checkout
**File**: `src/tree.cyr` (parse_tree), `src/commit.cyr` (materialize_target), `src/util.cyr` (ensure_dirs_for)
`parse_tree` accepts raw bytes between the mode-terminating space and the next `\0` as the entry name — zero validation. `materialize_target` calls `file_write_all(tpath, ...)`, `ensure_dirs_for(tpath)`, and `sys_unlink(cpath)` directly on attacker-controlled names. A malicious tree object with an entry named `../../../.ssh/authorized_keys` lands there during `sit clone <malicious>` / `sit pull` / `sit checkout`. This is the git CVE-2018-11235 / CVE-2019-1352 / CVE-2024-32002 shape.
**Impact**: Arbitrary file write/delete outside the repo, on any user who clones an attacker repo. With ref-writable paths, chains into persistent code execution once sit adds any `core.pager`-style config key.
**Fix**: `validate_tree_entry_name(name)` — reject empty, `.`, `..`, any `/` or `\` or `\0`, any `:` (NTFS ADS + drive-letter), any trailing space/dot (Windows), any case-fold-equal to `.sit` or `.git`, any HFS-ignorable codepoint (U+200C–U+200F, U+202A–U+202E, U+206A–U+206F, U+FEFF), any Windows reserved name (`CON`, `PRN`, `NUL`, `AUX`, `COM1-9`, `LPT1-9`). Call from `parse_tree` (reject the tree object entirely) AND from `materialize_target` (belt-and-suspenders). Also reject non-allowlisted modes — accept only `100644` (file) and `40000` (dir).

#### S-04 — Remote path trusted without symlink / shape validation (CVE-2023-22490 shape)
**File**: `src/wire.cyr` — remote_normalize, remote_objects_open, cmd_clone
`remote_normalize` only strips `file://` and hands anything else to `patra_open` or to filesystem reads. Three concerns:
1. `<repo_path>/.sit` or `<repo_path>/.sit/objects.patra` can be a symlink — patra dereferences it. Attacker repo's `.sit` → `/etc/shadow` or `/dev/zero` produces a misleading crash and may leak error-text content.
2. `remote_normalize` accepts `/home/victim/real-repo` — `sit clone` copies victim's own objects into attacker's target dir (not exfil from sit but a foot-gun when the URL comes from a third party).
3. `cmd_clone` does `ensure_dir(target)` + `syscall(SYS_CHDIR, target)` without checking whether `target` is already a regular file or a symlink to somewhere unexpected.
**Impact**: Data exfil via error-channel probing; user-surprise clobber; TOCTOU between `dir_list` emptiness check and chdir.
**Fix**: `lstat` both `<repo_path>/.sit` and `<repo_path>/.sit/objects.patra` before opening; refuse if either is a symlink. Reject remote URLs that don't begin with `file://`, `/`, or `./`. Refuse `cmd_clone` if resolved `realpath` of `target` and `source` share an inode (prevents self-clone). Add an ADR for the local-clone threat model.

### HIGH

#### S-05 — Config value newline injection (CVE-2023-29007 / CVE-2025-48384 shape)
**File**: `src/config.cyr` — config_file_set; also `src/wire.cyr` cmd_clone line where URL is written
`value` is `memcpy`'d into `key + " = " + value + "\n"` with zero sanitization. `sit remote add origin $'https://ok\nother.bad = payload'` injects a second config line. sit has no exec-shaped config keys today — so impact is "config corruption, surprising fetch/push targets" — but this becomes RCE the moment sit adds a `core.pager` / `core.editor` / `merge.driver` / `remote.<n>.receivepack` equivalent. CVE-2025-48384 is actively exploited in git for exactly this reason and is on CISA KEV.
**Impact**: Config state injection today; latent RCE tomorrow. Also: `sit clone <url>` stores `url` directly, so a URL with newlines attacks even without explicit `remote add`.
**Fix**: Reject `\n`, `\r`, `\0` in both key and value at the `config_file_set` boundary. Cap value length at 1024 bytes. Consider an allowlist of valid characters per key class.

#### S-06 — `alloc(size)` from unbounded `lseek(SEEK_END)` result
**File**: `src/index.cyr` (cmd_add), `src/util.cyr` (read_file_heap)
`size = syscall(SYS_LSEEK, fd, 0, SEEK_END)`; checked for `< 0` but no upper bound. `sit add /dev/zero` or a symlink-to-huge-file triggers a multi-GB allocation. `alloc(size + 1)` can also wrap if `size == INT64_MAX`.
**Impact**: OOM DoS, plus latent integer overflow.
**Fix**: Cap `size` at a sane maximum (say 1 GiB for `sit add`, 64 MiB for config files). Check `size + 1` for overflow before passing.

#### S-07 — LCS DP table integer overflow
**File**: `src/diff.cyr` — lcs_diff
`cells = (n1 + 1) * (n2 + 1)`; cap check against `16M` happens AFTER the multiply. Inputs where `n1 * n2` wraps produce a small `cells` that bypasses the cap, then `alloc(cells * 8)` under-allocates while the nested loop writes `n1 * n2` cells. Reachable via crafted blobs fed to `sit diff`, `sit show`, and the 3-way merge.
**Impact**: Heap corruption during diff of crafted large blobs.
**Fix**: Check `n1` and `n2` individually against `~4096` (`sqrt(16M)`) before multiplying.

#### S-08 — zlib decompression ratio unchecked (zip-bomb / parser amplification)
**File**: `src/object_db.cyr` (read_object), `src/wire.cyr` (db_object_read_decompressed)
`zlib_decompress` allowed to expand any single object up to 16 MiB. A commit body that decompresses near 16 MiB with a tiny on-disk footprint causes the `parse_commit_body` / `parse_tree` parsers to walk attacker-shaped bytes; a tree body of 16 MiB with 400,000 entries then drives materialize into O(N·path-len) I/O. No aggregate cap across objects in a single clone.
**Impact**: Per-object memory DoS; aggregate disk+memory DoS on clone; amplifies S-03's per-entry write.
**Fix**: (a) Validate the decompressed length against the framing header's declared length; (b) cap reachable-object count during clone (default 1M, configurable); (c) cap aggregate bytes written during `materialize_target` (default 10 GiB); (d) tighten `dec_cap = clen * 16` + single retry at 16 MiB.

#### S-09 — owl binary resolved from hard-coded writable paths
**File**: `src/object_db.cyr` — find_owl
Probes `/usr/local/bin/owl` → `/usr/bin/owl` → `/opt/owl/bin/owl` and execs the first match. No integrity check, no config override, no `$PATH` resolution. If an attacker gets write access to any of the three paths (package manager race, writable `/opt`), `sit owl-file` executes their binary as the sit user.
**Impact**: Arbitrary code execution via `sit owl-file`.
**Fix**: Accept explicit path via `SIT_OWL` env var or `core.owl.path` config. Refuse to exec if owl lives in a user-writable dir. Add `core.owl.disable` for full opt-out.

#### S-10 — `/tmp/sit-owl-<hex12>` predictable + world-readable + symlink-race
**File**: `src/object_db.cyr` — cmd_owl_file
`file_write_all` opens with mode 0644 (world-readable); path has only ~48 bits of entropy. Another local user can read blob contents during the owl run. The path is also vulnerable to a symlink-planted race between `file_write_all` and `exec_vec`.
**Impact**: Local info disclosure of object content; symlink attack to overwrite arbitrary user-writable files.
**Fix**: Use `$XDG_RUNTIME_DIR` (falls back to `$HOME/.cache/sit/` or `/tmp/sit-<uid>/` with chmod 0700). Open with `O_CREAT | O_EXCL`, chmod 0600. Reject if target already exists or is a symlink.

#### S-11 — Ref file content not hex-validated on read
**File**: `src/refs.cyr` (resolve_ref_name, read_head_ref), `src/wire.cyr` (read_remote_ref)
All ref readers take the first 64 bytes of the file and return them as "the hash" with no validation they're hex. Corrupt or attacker-crafted ref files yield garbage that flows downstream to `hex_decode` (which returns 0 silently), to SQL injection via S-01 (since the "hash" interpolates into `LIKE`), and to `write_head_ref` overwriting HEAD with garbage during pull.
**Impact**: Corrupt-ref → SQL-injection chain; silent corruption of HEAD.
**Fix**: Call `hex_is_valid(buf, 64)` (already exists in sigil) before returning. Treat invalid as "no such ref".

#### S-12 — Signing-key file written without `O_EXCL` (TOCTOU)
**File**: `src/sign.cyr` — cmd_key_generate
`file_exists(priv_path)` check; then `sys_open(..., O_WRONLY | O_CREAT | O_TRUNC, 0x180)`. If another local user plants a symlink at `~/.sit/signing_key` between the check and the open, the seed gets written through the symlink.
**Impact**: Local TOCTOU — attacker controls where the private key lands on disk.
**Fix**: Add `O_EXCL` to the open flags; reject on EEXIST. Also `lstat` the target first.

#### S-13 — `write_remote_tracking` stack buffer too small for long remote names
**File**: `src/wire.cyr` — write_remote_tracking
`var dir = alloc(128)` then memcpys `.sit/refs/remotes/<remote>` — when remote name exceeds ~110 chars, writes past the buffer into adjacent heap.
**Impact**: Heap overflow via a long remote name passed to `sit remote add`.
**Fix**: `alloc(18 + strlen(remote) + 1)`; audit all fixed-size `alloc(N)` sites for similar unchecked concat.

#### S-14 — Recursive `flatten_tree` / `walk_reachable_tree` without depth cap
**File**: `src/tree.cyr` (flatten_tree), `src/wire.cyr` (walk_reachable_tree)
Both recurse once per subtree. A crafted object set with N levels of nested subtrees blows the C stack. No depth counter.
**Impact**: Stack overflow on malicious remote content during `sit clone` / `sit pull` / `sit fsck`.
**Fix**: Iterative BFS with an explicit depth cap (e.g. 256) and an entry-count cap per tree (e.g. 100,000).

#### S-15 — `.sitignore` glob_match recursion unbounded
**File**: `src/index.cyr` — glob_match
Recurses for each `*` in the pattern. A pattern like `****...***a` with N stars and an M-char name is O(2^min(N,M)) stack depth. `.sitignore` is user-trusted today, but becomes attacker-controlled if synced via `sit pull`.
**Impact**: Stack overflow / CPU DoS on crafted `.sitignore` after pull.
**Fix**: Cap pattern length (e.g. 256 bytes), iterative glob_match (dynamic programming on positions).

### MEDIUM

#### S-16 — Filesystem mutation return values discarded
**File**: Multiple — `sys_unlink` after owl exec, `sys_unlink(cpath)` in materialize, `sys_unlink(".sit/MERGE_HEAD")`, `file_write_all` in materialize, `write_remote_tracking` return.
Silent failures leave stale tempfiles, stale `MERGE_HEAD` (future commits become unintended 2-parent merges), partial working trees, and ref desync.
**Fix**: Check every return; abort the command on failure with a clear error.

#### S-17 — `alloc` return value not null-checked in hot paths
**File**: Pervasive — object_db, tree, commit, merge, diff (only diff checks).
On OOM or freelist exhaustion, subsequent `memcpy(ptr + pos, ...)` writes to `0 + offset`. Cyrius has no language-level bounds checking.
**Fix**: Audit every `alloc()` call; propagate 0 consistently. Add an `alloc_or_die` helper for the cases where OOM should be fatal.

#### S-18 — `parse_author_line` timestamp integer overflow
**File**: `src/commit.cyr`
`ts = ts * 10 + (c - 48)` with no overflow check. A 20-digit timestamp in a crafted commit wraps.
**Fix**: Cap digit count at 19; saturate on overflow.

#### S-19 — `extract_sitsig` underflow if `body_len < 201`
**File**: `src/sign.cyr`
Inside the `line_len == 200` branch: `stripped_len = body_len - 201` — underflows to ~2^64 if `body_len < 201`. Not currently reachable given real commit shapes, but fragile.
**Fix**: Explicit guard `if (body_len < 201) return 0;` at the top.

#### S-20 — `hex_decode` result length unchecked at sitsig call sites
**File**: `src/sign.cyr` — extract_sitsig
Only checks `!= 0`. If sigil's `hex_decode` can return non-null with partial decode on invalid chars, `memcpy(out_sig, sig_bytes, 64)` reads past the returned buffer.
**Fix**: Validate hex chars with `hex_is_valid` first; require `hex_decode` to decode all bytes or fail.

#### S-21 — Log / cat-file output echoes attacker-controlled bytes unfiltered
**File**: `src/commit.cyr` — print_commit_header
Author identity written verbatim to stdout. Embedded `\r\n`, ANSI escape sequences, or carriage-return overwrite tricks let a malicious commit forge subsequent log lines or spoof commit metadata in a terminal.
**Fix**: Strip ASCII control chars (`< 0x20` except `\n`, plus `\x7f`) and ANSI escape introducers from identity bytes on display.

#### S-22 — `index_migrate_from_plaintext` accepts unbounded path length
**File**: `src/index.cyr`
Legacy plaintext index parser caps on record count but not per-line `plen`. 100 MB on one line → 100 MB allocation.
**Fix**: Cap `plen` at 4096 bytes.

#### S-23 — `cmd_clone` chdir into argv target without abs-path guard
**File**: `src/wire.cyr`
`target` is argv[3] or derived from URL's basename. No rejection of absolute paths. `sit clone <url> /etc/passwd` creates a dir there (if user has perms) and clones into it.
**Fix**: If target is absolute, require an explicit `--force-absolute` flag; otherwise treat as a relative-path-only operation.

#### S-24 — Patra handle + SQL-string leaks on error paths
**File**: `src/object_db.cyr`, `src/wire.cyr`
Allocated SQL-string buffers in `db_*` helpers are never freed. Patra handles close in the happy path but leak on several early-return branches.
**Fix**: Use `fl_alloc` / `fl_free` for short-lived query strings; refactor `read_object` to a single-exit pattern.

#### S-25 — `ensure_dirs_for` has a dead branch and is redundant with `ensure_parent_dirs`
**File**: `src/util.cyr`
`ensure_dirs_for` mkdirs each `/`-delimited prefix without the `i > 0` guard, so for absolute paths it attempts `sys_mkdir("", ...)`. Not exploitable; latent bug. The sole caller (`write_conflict_file`) could use `ensure_parent_dirs` instead.
**Fix**: Delete `ensure_dirs_for`; replace its caller.

#### S-26 — `refname_valid` missing; receive-side ref names from remote are trusted
**File**: `src/wire.cyr` — do_fetch, write_remote_tracking
Ref names discovered on a remote are never validated before being written into `.sit/refs/remotes/<remote>/<branch>`. A malicious remote advertising `../../../etc/cron.d/x` chains with S-02's write_remote_tracking primitive.
**Fix**: Same `refname_valid` as S-02, applied on the receive side.

#### S-27 — `materialize_target` ignores blob-read failures silently
**File**: `src/commit.cyr`
`read_blob_content` return value is implicitly checked (`if (== 0)`), but the caller produces a partial checkout with no warning on miss.
**Fix**: Emit a clear error and abort the entire materialize; leave the working tree in its prior state when possible.

### LOW

#### S-28 — `exec_vec` inherits full environment
**File**: `src/object_db.cyr` — cmd_owl_file
No scrubbing of `LD_PRELOAD` / `LD_LIBRARY_PATH` before execing owl. Not a cross-user boundary crossing today, but convention.
**Fix**: Minimal envp. When S-09 lands, include env scrubbing in the same change.

#### S-29 — Upward-directory `.sit` discovery is (correctly) not implemented — pin with an ADR
**File**: All entry points
sit requires `.sit/HEAD` in cwd — no upward walk. This is the right design and avoids CVE-2022-24765 (git's "trust any parent `.git/config`" bug). Lock it in with an ADR and a regression test that `sit status` in a subdirectory of a repo **fails**.
**Fix**: Write ADR 0003 — "sit does not search upward for `.sit/`".

#### S-30 — SHA-1 objects will never be accepted — pin with an ADR
**File**: `docs/adr/`
sit exclusively uses SHA-256. Document that legacy SHA-1 git objects will not be supported, even for interop, to prevent future downgrade attacks.
**Fix**: Write ADR 0004 — "sit is SHA-256 only".

#### S-31 — Patra-returned cstrings assumed NUL-terminated
**File**: Everywhere `patra_result_get_str` → `strlen` is chained.
If patra ever stores bytes without a terminator, `strlen` walks until it finds one (OOB read).
**Fix**: Use sized getter if/when patra exposes one; add a depth guard in the interim.

#### S-32 — Build_tree mode literals stored as string pointers (semantics audit)
**File**: `src/tree.cyr` — `store64(le, "100644")`
Stores a code-segment string pointer into an entry slot. Works because Cyrius string literals have program-lifetime, but depends on that invariant holding across Cyrius versions.
**Fix**: Confirm the invariant with Cyrius team (or switch to integer mode codes with a tiny format table).

## Performance findings

Numbering: `P-NN`. Effort scale: **S** (single function), **M** (multi-function refactor), **L** (algorithm change).

### Top-3 wins (collapse many findings together)

**Cache the patra object-DB handle process-wide** — collapses P-01, P-02, P-05, P-08, P-12, P-25. Every `read_object` / `write_typed_object` / `resolve_hash` currently calls `patra_init` + `patra_open` + `CREATE TABLE` + migration check + patra_close. `sit log` over 10K commits = 10K opens. `sit fsck` on a 10K-object repo = 10K opens. The wire-protocol walkers already use the right pattern (pass an opened handle); the fix is to apply that shape everywhere. **Effort: M** (cascades through 6+ callers).

**Batch `copy_objects` into a single transaction** — P-03. Current: 2-3 SQL ops per object (has/read/insert). For 1000-object fetch = ~3000 SQL round-trips, each WAL-synced. Fix: one `SELECT hash FROM objects WHERE hash IN (...)` to pre-filter, one `BEGIN` / N inserts / `COMMIT`. Push patra for a real `INSERT OR IGNORE` / `UPSERT` if not present. **Effort: S** for dedup; **M** for batching.

**Smarter decompression sizing + route LCS through freelist** — P-06 + P-15. Read the framing header's length prefix (`"blob 1234\0..."`) to size exactly, or adaptive-retry on `Z_BUF_ERROR`. Move the 128 MB LCS DP table to `fl_alloc` so diff-heavy commands don't permanently reserve bump memory. **Effort: M** for the sizing refactor.

### High-impact (scale with repo size)

- **P-04** — `walk_reachable_from_commit` decompresses every commit/tree in the history just to parse 1-2 header lines. Denormalize tree/parent hex into the patra schema or cache decompressed objects during the walk. **Effort: M**.
- **P-10** — `tree_find` is O(N) linear scan called in O(N) loops from `cmd_status`, `cmd_diff`, `materialize_target`. Build a hashmap once per command. **Effort: S**.
- **P-11** — `sit add` rewrites the entire index every call (via `rewrite_index`'s DELETE + N-INSERT). Push to patra's `UPDATE path = ? WHERE hash = ?` + `INSERT` fallback. **Effort: S**.
- **P-17** — Output fragments are separate `write(2)` syscalls (3000+ for a 500-line diff). Small stdout buffer with a flush at command end. **Effort: M**.
- **P-18** — `three_way_path_set` dedup is O(N²) via nested `streq`. Use the hashmap crate. **Effort: S**.

### Medium-impact

- **P-09** — `read_head_tree_entries` and `parse_index` called multiple times per command. Memoize. **Effort: S**.
- **P-13** — `is_ignored` is O(patterns × segments) per working-tree entry. Pre-classify patterns (plain / glob / suffix) and bucket. **Effort: M**.
- **P-20** — `sort_entries` is insertion sort (O(N²)). Push to `ORDER BY path` in the `parse_index` query. **Effort: S**.
- **P-22** — `parse_tree` allocates 3 heap strings per entry. Add a zero-copy variant for read-only walkers. **Effort: M**.
- **P-24** — `write_typed_object` does SELECT-before-INSERT for dedup. Collapse to patra's UPSERT when available. **Effort: S**.

### Lower-impact / backlog

- **P-14** — LCS cap is 4000×4000 lines; larger files get "too large" with no fallback. Implement Myers' O((N+M)D) for large-file diffs. **Effort: L**.
- **P-16** — `cmd_fsck` re-hashes via SHA-256; dominated by sigil throughput. Flag sigil's roadmap; add a `sit fsck --fast` that only checks framing.
- **P-07** — Bump allocator grows unbounded during long ops. Per-command arena reset or systematic move to `fl_alloc` for short-lived buffers. **Effort: L**.
- **P-19**, **P-21**, **P-23** — redundant `strlen` in loops, mode-string re-scans, `LIKE` vs range-bound prefix scan. Grouped micro-wins, address during the next perf-focused minor.

### Baseline (captured 2026-04-24 pre-P(-1))

SHA-256: 64B/14µs · 1KB/113µs · 64KB/6.3ms (~10 MB/s ceiling, sit-side `sit add` bottleneck).
zlib compress: 1KB/163µs · 64KB/1.5ms. zlib decompress: 1KB/39µs · 64KB/378µs.
See [`docs/benchmarks/2026-04-24-baseline.md`](../benchmarks/2026-04-24-baseline.md).

Add new benchmarks after fixes: `patra_open/close` round-trip (baseline target for P-01 win), 1000-object fetch end-to-end (P-03 target), `sit log` on a 1000-commit history (P-05 target).

## Post-v0.6.0 findings from benchmark run

### S-33 — `sit status` SIGSEGV on 100-commit / 100-file repo — RESOLVED in v0.6.1
**Severity**: CRITICAL (found 2026-04-24 during v0.6.0 git-vs-sit bench run)
**Reproducer**: `scripts/benchmark.sh` `bench_status`, or manually: init repo + loop 100× (touch file; `sit add`; `sit commit`); run `sit status`. Exit 139 (SIGSEGV). Threshold appears around 20-30 files in history but is intermittent — smaller repos sometimes also crash on repeated invocation.
**Impact**: `sit status` unusable at realistic repo scale. Blocks the `bench_status` and `bench_clone` benches (clone materializes a 100-commit tree which hits the same crash). Silent data / UX regression depending on when the crash occurs.
**Resolution (v0.6.1)**: Two stacked upstream bugs, neither matching the original candidate root causes:
- **cyrius stdlib `alloc` grow-by-1MB undersize** — the 16 MiB retry buffer in `read_object` returned a pointer past the brk; first tail-write SIGSEGV'd. Fixed in cyrius 5.6.34, picked up via the `cyrius.cyml` pin bump from 5.6.25 → 5.6.35. Writeup: [`docs/development/issues/archived/2026-04-24-cyrius-stdlib-alloc-grow-undersize.md`](../development/issues/archived/2026-04-24-cyrius-stdlib-alloc-grow-undersize.md).
- **sankoch zlib compress/decompress asymmetry** — caused `read_object` to fall into the 16 MiB retry path in the first place by failing the first decompress on ~20% of objects at scale. Fixed in sankoch 2.0.3, picked up via the `cyrius.cyml [deps.sankoch].tag` bump from 2.0.1 → 2.0.3. Independent of the alloc bug — proven by re-running the fixture across just the cyrius bump (bit-for-bit identical bad-object set). Writeup: [`docs/development/issues/archived/2026-04-24-read-object-unreadable-at-scale.md`](../development/issues/archived/2026-04-24-read-object-unreadable-at-scale.md).

The original candidate root causes (S-17 alloc null deref, P-07 bump exhaustion) were both wrong. Lesson for the next P(-1) pass: when triaging crashes that involve `alloc()`, instrument the allocator's growth path before guessing — the grow-by-1MB undersize signature (SEGV_MAPERR ~Nx MiB past the brk target) is unmistakable on strace and would have pointed straight at cyrius stdlib in minutes.

## Methodology notes for next P(-1) pass

What worked:
- Three parallel agents (security internal / CVE external / perf) completed in ~3.5 minutes each and produced non-overlapping findings.
- Targeted manual re-read of highest-risk paths (cmd_clone, parse_tree) caught S-03 before the agents reported it — worth doing in parallel rather than waiting.
- Baseline benchmark saved before any fixes — gives the post-review bench something to compare against.

What to improve:
- No fuzz harness exists for malicious tree / commit bodies. Add these before the v0.5.2 security patch so the validator has real coverage.
- The performance agent surfaced several findings that need measurement confirmation (e.g. the per-call patra open cost is asserted but not benchmarked). Add a patra-open micro-benchmark before fixing so we know the win size.
- No explicit test for the case where `patra_open` is called concurrently — do we have a race between the sit binary and a patra WAL recovery? Flag for patra.
