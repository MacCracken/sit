# Security + correctness audit — 2026-06-13 (v0.9.0 closeout)

The v0.9.0 stabilization pass before the v1.0.0 cut. Scope: the v0.8.x feature
additions (`log --graph`, `--depth` shallow clone, `merge_base` full-DAG LCA,
`fsck --prune`) plus a whole-tree dead-code / lint / security re-scan. An
independent adversarial review agent walked the four new function groups; its
findings are recorded and resolved below.

## Findings

| # | Severity | Area | Status |
|---|----------|------|--------|
| 1 | HIGH | `_graph_join_connector` seam index underflow (`buf-1` OOB write) | **Fixed** |
| 2 | HIGH→informational | `_graph_merge_connector` seam underflow | **Guarded** (unreachable, defended) |
| 3 | HIGH | `fsck --prune` deletes reachable objects behind a corrupt interior object | **Fixed** |
| 4 | LOW | `find_merge_base` returns 0 on a cyclic/corrupt store | **Fixed** |

### 1 — HIGH: `_graph_join_connector` seam underflow (memory safety)

`src/commit.cyr`. The join connector wrote `store8(buf + 2*rcol - 1, 47)`. When
the collapsing lane is column 0 (`rcol == 0`) the index is `-1`, an
out-of-bounds write to `buf-1`. Reachable from `sit log --graph` whenever a
merge's **second** parent (the feature lane) is *strictly* newer than the first
(the main lane): the feature lane emits first, then main's lane (column 0)
collapses. Common in real history (commit on main, then commit on a
feature branch off an older point hours later).

**Fix**: guard `if (rcol > 0)` before the seam write — a lane-0 collapse has no
column to its left, so no `/` is drawn. Verified: a diamond where the feature
commit is forced strictly-newer now renders cleanly and exits 0 (regression test
`tests/integration/run.sh` § "log --graph lane-0 join"). Documented cosmetic
note: the lane-0 collapse renders `  |` rather than git's `|/` (a known
simplification of the column-shifting sit deliberately doesn't do).

### 2 — `_graph_merge_connector` seam (same bug class, unreachable)

The merge connector has the same `2*k - 1` seam math. Analysis shows `base >= 1`
is invariant there (the join branch that could empty the lane set requires ≥2
lanes, leaving ≥1 after removal), so `k` starts at ≥1 and never underflows.
Guarded `if (k > 0)` anyway as defense-in-depth so the invariant is explicit.

### 3 — HIGH: `fsck --prune` data loss behind a corrupt object

`src/object_db.cyr`. `fsck_walk_reachable` silently skips any object that fails
`read_object` (no `else`), so it never enqueues that object's children. A single
corrupt interior commit/tree therefore orphans its entire subgraph into the
`dangling` set — and `--prune` ran **before** the `bad > 0` exit check, so it
would permanently delete genuinely-reachable objects (no reflog to recover).

**Fix**: two guards before the prune loop — (a) refuse `--prune` when `bad > 0`
(any unreadable/corrupt object), and (b) refuse when every object is unreachable
(`dangling == total`, the signature of a corrupt/missing HEAD that would
otherwise wipe the whole store). Mirrors the existing MERGE_HEAD guard.

### 4 — LOW: `find_merge_base` on a cyclic store

`src/merge.cyr`. In a well-formed DAG the redundancy reduction always leaves a
maximal survivor, so `best` is set. A corrupt parent cycle could make
`is_ancestor` report mutual ancestry, marking every candidate redundant →
`best == 0` → "no common ancestor" despite candidates existing. **Fix**: fall
back to the first candidate when `best == 0` (only reachable on a corrupt store).

## Dead-code audit

DCE (`CYRIUS_DCE_VERBOSE`) flags 7 sit-defined functions unreachable from the
CLI `main()`:

- `sit_repo_open` / `sit_repo_close` / `sit_diff_path` — **intentional**: the
  public API (ADR 0009), dead in the CLI binary but the live `dist/sit.cyr`
  library export consumed by owl. Keep.
- `build_commit` / `build_merge_commit` — **intentional** thin wrappers (the
  `*_signed` variants are the real builders; see CLAUDE.md). Keep.
- `_ssh_handle_batch_known` / `_ssh_handle_carry` — **removed**: getters for
  reserved SSH-handle slots with no readers. The zero-init setters stay (they
  keep the reserved layout defined for the future pipelining/batch work).

Remaining unreachable-from-binary floor: the 5 intentional functions above.

## Refactor

The `read_object → parse_commit_body → iterate out+48 parents` boilerplate had
accreted across `is_ancestor` (v0.8.7) and both BFS loops in `find_merge_base`
(v0.8.13) — three pure parent-graph walks past the rule-of-three. Extracted
`commit_parents_of(hex)` (`src/commit.cyr`), removing ~30 lines. The other
parent walkers (`walk_reachable_phased`, `print_graph`, `cmd_fsck`) also need
the tree / timestamp / body, so they keep their inline reads — a heavier shared
iterator wasn't worth the risk pre-1.0; noted as a post-1.0 opportunity.

## Mechanical security re-scan (clean)

- **No `sys_system`** anywhere in `src/`.
- **SQL interpolation**: every `WHERE hash = '<hex>'` / `LIKE '<prefix>%'` site
  validates the value as hex first (`hex_prefix_valid`) — including the new
  `db_object_delete` (64-char hex, into a 128-byte buffer holding a 100-byte
  query). No injection surface.
- **Buffers**: the v0.8.x fixed buffers (graph grid `2*nl-1`, shallow file
  `n*65`, merge/delete SQL `128`) check out; the only OOB was finding 1/2 (now
  guarded).
- **exec/execve**: `cmd_owl_file`'s `exec_vec` and `wire_ssh`'s `sys_execve`
  are the pre-existing argv-controlled paths (CVE-2017-1000117 defenses intact).

## Result

180 unit + 33 integration assertions, fuzz clean (6 harnesses), bench flat vs
the v0.8.12 baseline, lint clean. DCE binary 2.204 MB. Verdict: **ship** — the
three HIGH findings are fixed and regression-tested; the codebase is clean for
the v1.0.0 cut.
