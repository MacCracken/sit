# Benchmark snapshot — 2026-04-24 (v0.6.0)

Captured after the P(-1) audit fixes landed. Pairs with [`2026-04-24-baseline.md`](2026-04-24-baseline.md) (pre-audit).

**Host**: Linux x86_64, cyrius 5.6.25, sit 0.6.0

## sigil SHA-256 throughput

| Input size | Time/op | Implied throughput |
|-----------:|--------:|-------------------:|
| 64 B       | 10 µs   | ~6.4 MB/s |
| 1,024 B    | 90 µs   | ~11.4 MB/s |
| 65,536 B   | 5.443 ms| ~12.0 MB/s |

Slight improvement vs baseline (14µs → 10µs at 64B) — likely run-to-run variance, not a code change. Ceiling is still ~12 MB/s for large blobs; still dominates `sit add` on anything non-trivial. Flagged for sigil's roadmap.

## sankoch zlib

| Op | 1,024 B | 65,536 B |
|---|--------:|---------:|
| compress   | 142 µs | 1.310 ms |
| decompress |  35 µs |   355 µs |

Marginal improvement vs baseline; no change expected (validators don't touch zlib).

## Patra object-store hot path

| Bench | Time/op | Notes |
|-------|--------:|-------|
| `patra-open-close` | **19 µs** | Per-call cost that `read_object` / `write_typed_object` / `resolve_hash` pay on every invocation. `sit log` on a 10K-commit history spends ~190 ms in pure DB lifecycle before any work. This is the baseline the v0.6.x DB-handle-caching refactor will collapse (P-01). |
| `copy-objects-100` | **53 µs total = 530 ns/row** | 3 SQL ops per row (has + read + insert). Batching into a single transaction (P-03) should cut this 2-3× based on patra's WAL cost per commit. |

## Commit walk cost

| Bench | Time/op | Notes |
|-------|--------:|-------|
| `commit-parse+iso8601` | **1 µs** | Pure-CPU per-commit cost during `sit log` walk. Negligible compared to the `patra-open-close` tax above — confirms P-05's finding that the DB cost dominates, not the parsing. |

## Signing primitives

| Bench | Time/op |
|-------|--------:|
| `ed25519-sign`   | **1.203 ms** |
| `ed25519-verify` | **7.68 ms** |

Verify is ~6× slower than sign. Expected for ed25519 (verify does two scalar multiplications vs sign's one). `sit verify-commit` on a 1000-commit signed history would take ~8 seconds of pure ed25519. `sit show` / `sit log` decorate every commit with a signature check where present — something to keep an eye on as signed histories grow. Flag for sigil's roadmap.

## Validator throughput

| Bench | Time/op | Notes |
|-------|--------:|-------|
| `refname_valid-good`          | **319 ns** | `git check-ref-format` grammar walk. Called every ref write. |
| `tree_entry_name_valid-good`  |  **97 ns** | Every tree entry during parse + materialize. |
| `tree_flat_path_valid-good`   | **718 ns** | Called every entry during `materialize_target`. Slowest of the four because it allocs + copies a buffer per '/'-separated segment. |
| `hex_prefix_valid-64char`     | **207 ns** | Called every `resolve_hash` + every ref-file read. |

Validators are cheap (sub-microsecond) but not free. At clone time for a 10K-entry tree, that's ~7 ms in `tree_flat_path_valid` alone — well under 1% of the overall clone budget but worth knowing. Post-0.6.0, consider rewriting `tree_flat_path_valid` to avoid per-segment allocation (single pass, no scratch buffer).

## Deferred benches (queued for v0.7.0)

Per audit recommendations, these need larger fixtures or separate algorithm work:

- **LCS diff** at 100×100 / 1000×1000 / 4000×4000 line counts — shows the cost curve and the 16M-cell cliff. Motivates the Myers O((N+M)D) fallback on the roadmap.
- **`glob_match`** against 10 / 50 / 200-pattern `.sitignore` files — baseline for the P-13 pattern pre-classification refactor.
- **`hash_file_as_blob` end-to-end** (`lseek` + `read` + frame + SHA-256) — measures the true `sit add` floor on realistic-sized inputs (1 KB / 64 KB / 1 MB). Motivates the sigil throughput story.

Land alongside v0.7.0 when the algorithm work that justifies them comes into scope.
