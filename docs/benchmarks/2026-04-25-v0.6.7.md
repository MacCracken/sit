## Where the win is

**Clone -22%**. `walk_reachable_*` was decompressing every commit + tree to discover them, which internally called `db_object_read_raw` to pull the compressed bytes from `src_db`. Then `copy_objects` re-read the same compressed bytes (calling `db_object_read_raw` a second time) for the insert into `dst_db`. Every commit + tree was paying for two source-DB reads instead of one.

P-04 fix: `db_object_read_both(db, hex, raw_out, deco_out)` returns BOTH the compressed bytes (formerly thrown away) and the decompressed view. `walk_reachable_tree` and `walk_reachable_from_commit` now stuff the raw bytes into a `raw_cache` map keyed by hex. `copy_objects` checks the cache first; cache-miss (only blobs, since walk doesn't visit them) falls back to `db_object_read_raw`.

Concrete savings on the 100-commit / 100-file fixture:
- Before: walk reads 100 commits + 100 trees = 200 source reads. Copy reads 100 commits + 100 trees + 100 blobs = 300. **Total: 500 source reads.**
- After: walk reads 200 source reads (same). Copy reads 100 blobs (cache miss only). **Total: 300 source reads.**
- 40% fewer source SQL ops → 22% wall-clock improvement on clone.

The win is bigger than the naive "redundant read = patra page fetch only" model suggests because each saved read also avoids: SQL string parse, patra B+ tree lookup walk, and the result-set buffer setup. All three add up to ~150-200µs per saved read; 200 saved reads × ~150µs = ~30-40ms, matching the observed 47ms wall-clock savings.

## What didn't move and why

Same dep-side bottleneck story as previous releases:

- `add-*`: dominated by sankoch zlib + sigil hash; both filed on those repos' roadmaps.
- `status-100files`: dominated by 100× sigil SHA-256 + 100× file open/read.
- `log-100commits`: P-04 helps walk_reachable, but `cmd_log` doesn't use walk_reachable — it walks the commit chain via `read_object` per commit. Already cached at the patra-handle level (v0.6.4).
- `diff-edit`: bottleneck is LCS DP table allocation + zlib_decompress. P-15 (LCS via fl_alloc) is queued but won't move wall-clock.

## Next ship targets

Per [the roadmap](../development/roadmap.md), remaining sit-side perf items, ranked by projected wall-clock impact:

- **P-17** buffered stdout — `diff-edit` does ~3000 `write(2)` syscalls; theoretical ~3ms savings on 13.7ms = ~22%. Actual benefit depends on syscall cost.
- **P-15** LCS DP table to `fl_alloc` — pure memory hygiene, no wall-clock movement. Worth doing for hygiene; not urgent.
- **P-06** smarter decompression sizing — modest. Avoids the worst-case 16 MiB retry alloc on hostile inputs; needs partial decompression to read the framing length prefix; medium complexity for small win.

The next significant `clone` improvement is **patra 1.7.0 `INSERT OR IGNORE`** landing — would drop the inner `db_object_has` and project ~30% on top of v0.6.7 (clone toward ~120ms = ~8x git).
