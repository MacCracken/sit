## What changed and why nothing moved (yet)

P-17 swapped 200+ direct `syscall(SYS_WRITE, STDOUT, ...)` call sites across nine source files to a single buffered `stdout_write(data, len)` helper backed by a 64KB heap buffer. The buffer auto-flushes on full and is explicitly flushed in `src/main.cyr`'s trailer before `SYS_EXIT`. STDERR (`eprintln`) stays direct — error output should be immediate even when the process is being killed mid-command.

`write_sanitized` (used for attacker-controlled commit body fragments under S-21) was a one-syscall-per-character offender on top of being a buffer-bypass. Rewritten in the same change to build the sanitized output into a heap buffer in one pass and emit via a single `stdout_write` call.

**Why the bench didn't move:** the audit's "3000+ writes for a 500-line diff" estimate was for very large diffs with many hunks. The bench's `diff-edit` fixture is a 500-line file with ONE changed line — actual output is one hunk ≈ 30 writes. 30 syscalls × ~1µs each = ~30µs — already well under the noise floor of the 13ms `diff-edit` budget.

The win is real at scale. Ad-hoc test on a 2000-line file with 1000 changes (45KB of diff output): ~185ms median. Without buffering that'd be ~1000 separate `write(2)` syscalls; with buffering it's one. Saves ~1ms+ on a 185ms total — visible if benched, not currently in the harness.

Structural benefit of the change is real even when wall-clock isn't moving: every command now flushes once at exit instead of streaming bytes through the kernel one syscall at a time. Reduces syscall pressure on the system as a whole, lower context-switch cost, cleaner output ordering (the buffer guarantees in-order writes that ad-hoc syscall mixes did not — caught a real ordering bug during this work where `write_sanitized` was emitting the author identity BEFORE the "Author: " prefix that should have preceded it; fixed in the same change).

## What's still unmoved and why

Same dep-side bottleneck story:

- `add-*`: dominated by sankoch zlib + sigil hash; both filed on those repos' roadmaps.
- `status-100files`: dominated by 100× sigil SHA-256 + 100× file open/read.
- `log-100commits`: bound by per-commit decompress + parse, both already at the implementation ceiling sit can move alone.
- `clone-100commits`: P-04 walk-cache landed in v0.6.7 (-22%); the next significant win is patra 1.7.0 `INSERT OR IGNORE` (would drop the inner `db_object_has` and project ~30% on top of v0.6.8).

## Next ship targets

Sit-side perf items remaining:

- **P-15** LCS DP table to `fl_alloc` — pure memory hygiene, no wall-clock movement. Hygienic but not a release driver alone; could fold into the next bug fix release.
- **P-06** smarter decompression sizing — modest, medium complexity. Avoids the worst-case 16 MiB retry alloc on hostile inputs; needs partial decompression to read the framing length prefix.

Honest read: sit-side perf wins for the standard 100-file fixture have plateaued. Cumulative `clone -32%`, `log -17%` from a half-dozen small changes. The remaining sit-side items target either edge cases (P-06 hostile inputs) or hygiene (P-15 bump-heap pressure) — neither moves the headline numbers.

The next significant headline movement requires upstream work: patra 1.7.0 `INSERT OR IGNORE` for clone, sigil SHA-256 throughput for status/add, sankoch zlib throughput for add. All filed on those repos' roadmaps.
