## What changed

### P-20 â€” `ORDER BY path` in `parse_index`'s SELECT

`src/index.cyr:parse_index` now passes `SELECT path, hash_hex FROM entries ORDER BY path` instead of an unordered SELECT. Patra's SQL parser handles ORDER BY since at least 1.6.0; sit just wasn't using it.

Downstream callers (`cmd_commit`, `cmd_status`, `cmd_diff`, the index-mutation paths in `cmd_add` / `cmd_rm` / `cmd_reset`) all run `sort_entries(entries)` after `parse_index` â€” an insertion sort that was **O(NÂ²)** on unsorted input. With ORDER BY at the patra layer, the entries arrive sorted; insertion sort then runs in **O(N)** (one pass that finds each element already in place).

Concrete savings (per `parse_index` call):
- 100 entries: ~5000 streq comparisons â†’ ~100 comparisons. Saves ~50Âµs.
- 1000 entries: ~500K comparisons â†’ ~1000. Saves ~5ms.
- 10000 entries: ~50M comparisons â†’ ~10000. Saves ~500ms.

The bench fixture uses 100 entries â€” the saving is well under the noise floor of the surrounding work. Real win at monorepo scale; not visible at the synthetic bench. Same shape as v0.6.6 (P-10 + P-18) and v0.6.8 (P-17): structural improvement that lands cleanly but doesn't move the small-fixture wall-clock.

### Investigated + reverted: multi-insert transactions in `cmd_commit` and `rewrite_index`

After v0.6.10's BATCH-mode revert, the open question was whether wrapping the multi-insert sequences in `cmd_commit` (tree + commit object writes) and `rewrite_index` (DELETE + N INSERTs) in explicit `patra_begin` / `patra_commit` pairs would amortize fsync the way `copy_objects` does (v0.6.5 P-03, the change that gave clone its biggest single-release win).

Implemented both. **A/B measured on a 50 `sit add` + `sit commit` cycle workload**:

- pre-v0.6.11 (no txn wraps): 230 / 232 / 232 ms across 3 runs
- post-v0.6.11 (with txn wraps): 251 / 252 / 241 ms across 3 runs

**5-10% regression**, consistent across runs. Reverted before shipping.

Why the wraps don't help here when they helped on `copy_objects`:

| Workload | Inserts per txn | Fsync savings | Setup overhead | Net |
|---|---:|---:|---:|---|
| `copy_objects` (clone) | ~300 | huge (300 â†’ 1) | one ~30Âµs setup | **wins** (-15% in v0.6.5) |
| `cmd_commit` | 2 (tree + commit) | tiny (2 â†’ 1) | one ~30Âµs setup | **loses** by ~10Âµs |
| `rewrite_index` | 1â€“50 (varies) | small-to-medium | per-call setup | break-even at Nâ‰ˆ30; loses for N<30 |

On modern SSDs, the per-insert fsync cost is small enough (kernel batches dirty-page flushes) that patra's transaction setup/teardown (lock + header read + WAL start + commit + header write + unlock) exceeds the savings unless the batch is large. `copy_objects` is large enough; `cmd_commit` and `rewrite_index` aren't.

This would likely flip on rotating disks or busy SSDs where per-insert fsync is more expensive â€” but optimizing for the slow case via a regression on the fast case isn't a defensible tradeoff. Filed under "won't ship without per-host benchmarking + a configuration knob, neither of which is justified yet."

The investigation lives in this snapshot doc plus the v0.6.11 CHANGELOG so future work doesn't waste time relearning it.

## What remains

Sit-side perf items still legitimately on the table:

- **Larger bench fixtures** (1000+ files) to validate the v0.6.6 hashmap wins, v0.6.8 buffered stdout wins, and v0.6.11 P-20 sort win. Requires the bench harness to grow a "scale" mode that runs less frequently due to runtime cost. Worth doing when there's a real consumer pushing those scales OR when we want defensible numbers for a public README.
- **P-13** `is_ignored` bucket pre-classification (effort: M). Helps repos with large `.sitignore`. Worth doing if any consumer hits it.
- **P-22** `parse_tree` zero-copy variant (effort: M). Eliminates 3 heap allocs per tree entry. Modest win; bigger at scale.

Everything bigger is dep-side:
- **patra `INSERT OR IGNORE` flag on `patra_insert_row`** would let sit drop the inner `db_object_has` (still asks one SELECT per object during clone)
- **sigil 2.9.2** SHA-256 throughput investigation
- **sankoch** DEFLATE throughput investigation
