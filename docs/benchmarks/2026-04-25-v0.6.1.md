# Benchmark snapshot — 2026-04-25 (v0.6.1)

Captured after the S-33 dep-bump release. First snapshot to include `status-100files` and `clone-100commits` rows — those were disabled in v0.6.0 because the SIGSEGV blocked them.

**Host**: Linux 6.18.22-1-lts x86_64, cyrius 5.6.35, sankoch 2.0.3, sit 0.6.1, git 2.53.0

### What's new vs the v0.6.0 snapshot

- **`status-100files`**: 7.08 ms / 1.80x git. Previously 0/0 — the row didn't exist because `sit status` SIGSEGV'd on the fixture. Below 2× git for the first time. Floor is dominated by 100× `read_object` for the HEAD tree's blob hashes; once the v0.6.x DB-handle-caching refactor lands (P-01) this should drop substantially.
- **`clone-100commits`**: 245 ms / 15.87x git. Also previously 0/0 for the same reason. The 16× ratio is the per-call patra open tax cubed: clone reads every reachable object via `read_object`, each of which opens + closes a patra handle. P-01 is the fix. Until then, this row is the v0.6.x perf-arc target.

Other rows are within run-to-run noise of the v0.6.0 snapshot — no source changes in v0.6.1, so any movement is environmental.

### Patra object-store hot path

| Bench | Time/op | Notes |

## What's NOT in this snapshot

- The S-33 fixture itself (100 commits + status + fsck) — it lives in `scripts/benchmark.sh` already, not in the standalone bench binary. The end-to-end status/clone numbers above are the relevant proof-of-life.
- LCS / glob / `hash_file_as_blob` — still queued for v0.7.0 per the v0.6.0 snapshot's "Deferred benches" section.

## Takeaways for v0.6.x perf arc

The clone 16× ratio is the headline gap. It's a single architectural fix (P-01 — cache the patra DB handle) that collapses ~5 of the top-10 hot paths simultaneously. Anything else (LCS algorithm, batched inserts, smarter decompression sizing) is downstream and should wait until P-01 lands so we're optimizing against the right baseline.
