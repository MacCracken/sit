## What changed

### cyrius 5.6.35 → 5.6.40

Five toolchain patches. None target a bottleneck the sit bench is sensitive to. Picked up for general hygiene + to stay close to head — useful if any future cyrius change wants a recent toolchain.

### patra 1.6.0 → 1.8.3 — three new things; sit picked up two

Patra shipped three things between 1.6.0 and 1.8.3:

1. **1.6.1 — `patra_result_get_str_len(rs, row, col)` sized accessor.** Bound-walks the COL_STR_SZ slot for a STR column. **Picked up by sit in v0.6.10**: replaces the v0.6.3 `strnlen(path_str, 256)` workaround in `parse_index` (S-31). One callsite swap, helper deleted. No bench movement (the workaround was always O(scan-up-to-256) — same complexity as the new accessor).

2. **1.7.0 — `INSERT OR IGNORE INTO` SQL syntax.** Native dedup at insert time; the consumer can drop a `SELECT exists?` round-trip per insert. **Not picked up by sit in v0.6.10**: sit's object-store inserts go through `patra_insert_row` (the only path that handles BYTES columns), not through SQL strings. patra's `INSERT OR IGNORE` is SQL-level only — `patra_insert_row` doesn't expose an `or_ignore` flag yet. Filed informally; sit will pick this up if patra grows the programmatic shape.

3. **1.8.x — WAL group commit (`patra_set_sync_mode(db, PATRA_SYNC_BATCH)`).** Defers fdatasync from per-mutating-exec to every-64-writes (`PATRA_BATCH_FLUSH_N`) or explicit `patra_flush()`. **Investigated and NOT enabled in sit v0.6.10** — see "BATCH mode investigation" below. Returned BATCH mode call sites to default FULL.

### S-31 closeout

The 2026-04-24 audit flagged "patra-returned cstrings assumed NUL-terminated" as a defense-in-depth gap. v0.6.3 shipped a `strnlen(s, 256)` workaround in `src/util.cyr` and used it in `parse_index`. That same release's CHANGELOG noted the upstream-fix-when-patra-ships-a-sized-getter path. Patra 1.6.1 shipped that getter. v0.6.10 swaps the workaround for the native API and deletes the `strnlen` helper (no other consumers).

## BATCH mode investigation (and why we reverted)

Initially set `patra_set_sync_mode(_object_db_cached, PATRA_SYNC_BATCH)` and the same on the index handle. Re-bench: `clone-100commits` 170.92 ms vs v0.6.9's 172.64 — within run-to-run noise. **No measurable improvement.**

Why nothing moved:

- **`copy_objects` already uses explicit `patra_begin` / `patra_commit`** (v0.6.5 P-03). Inside an explicit transaction, writes are already buffered to WAL with no per-write fsync; the fsync happens at commit. BATCH mode adds nothing on top.
- **`commit` does 3 inserts (blob + tree + commit)**, each currently its own implicit transaction. BATCH mode would coalesce them, but only across the every-64-writes auto-flush boundary. 3 < 64; the cumulative pending counter never trips an auto-flush within a single `sit commit`.
- **The cached patra handle never closes** — fd dies with the process. patra_close would flush pending writes; we don't call it. So in BATCH mode, writes that haven't tripped the every-64 threshold sit in patra's WAL pending buffer until... never (within sit's process lifetime).

Why it was a durability regression to keep:

If sit crashes (or the host loses power) between two `sit commit`s in BATCH mode, the most recent commits sit in the kernel writeback window with no fdatasync — they can be lost on power loss. With FULL mode (default), every commit is durable on return.

Net: **no perf gain, real durability cost**. Reverted before shipping. Revisit when sit grows explicit `patra_flush()` calls at command exit (clean durability boundary at the right time, and BATCH gets to amortize multi-insert commands like `commit`).

## What we didn't do — and how to do it later

Two paths exist for clone to drop further once a future change lands. Neither shipped in v0.6.10:

- **Wrap multi-insert commands in explicit transactions + flush at exit.** `cmd_commit`'s 3-insert sequence is the obvious candidate; `cmd_add` if it ever batches. Combined with BATCH mode this would cleanly collapse fsyncs without the durability regression. Effort: small. Filed for v0.6.11 consideration if the user wants further perf vs simplicity tradeoff.
- **Programmatic INSERT OR IGNORE in `patra_insert_row`.** Would let sit drop the inner `db_object_has` check inside `db_object_insert_raw` (the one that's still doing 1 SELECT per object inside `copy_objects`). Filed informally on patra; would need a patra release to grow the API.
