## Where the win is — and isn't

**Win:** `sit log` is ~17% faster on a 100-commit history. Walking the commit chain is exactly the workload patra-handle-caching was designed for — N read_object calls in a tight loop, each previously paying patra_init + patra_open + CREATE TABLE + loose-migration check + patra_close. With the cache, all of that runs once per process; the loop is pure bytes-read + zlib-decompress + commit-parse. Higher commit counts amortize the same fixed-cost win; expect proportionally larger savings on 1K- or 10K-commit walks (not measured here — the bench fixture is 100 commits).

**Not a win:** `clone-100commits` and `status-100files` did not move (within run-to-run noise). This is expected — their bottlenecks are downstream of the patra open/close cost:

- **Clone (245 ms)**: dominated by per-object zlib_decompress (sankoch) and per-file file_write_all during materialize. `walk_reachable_from_commit` does use `read_object` per object, so the cache helps that part — but the wins are absorbed by the much larger decompression + I/O budget. Real clone wins need P-03 (batch copy_objects in a single transaction) or P-06 (smarter decompression sizing) — both queued for a future v0.6.x release.
- **Status (7 ms)**: dominated by 100× `hash_file_as_blob` (file read + SHA-256 hash). Sigil's SHA-256 throughput is ~12 MB/s and `sit add` / `sit status` are sigil-bound; the patra cache touches a single-digit percentage of the time. Real status wins need sigil throughput improvements (sigil's roadmap, not sit's).
- **Add (`-1MB` 210 ms)**: 200ms is just `zlib_compress(1 MB)` + `sigil hash` of the same 1 MB. This is the language-level ceiling, not a sit refactor target.

## What changed (architecture)

- **Process-wide cached handle for `.sit/objects.patra`** (`get_object_db()` in `src/object_db.cyr`). Lazy-open on first call; lazy `object_db_migrate_from_loose` on first call. fd dies with the process (patra_close just frees in-process buffers and closes the fd — no WAL flush — so skipping it at exit is safe).
- **Process-wide cached handle for `.sit/index.patra`** (`get_index_db()` in `src/index.cyr`). Same shape.
- `read_object`, `write_typed_object`, `resolve_hash`, `cmd_fsck` (object_db.cyr) and `parse_index`, `rewrite_index` (index.cyr) all migrated to the cache; their previous `var db = X_open(); ... patra_close(db);` pattern is gone — single-exit refactor (S-24) fell out for free.
- `do_fetch` and `do_push` (wire.cyr): the LOCAL DB end (dst_db in fetch, src_db in push) now uses the cache. The REMOTE end stays per-call because each operation addresses a different file.
- **S-24 fl_alloc swap**: SQL-string buffers in `read_object`, `resolve_hash`, `write_typed_object` switched from `alloc_or_die` (bump heap, lives forever) to `fl_alloc` + `fl_free` (mmap'd, freed after each query). This trims per-query bump-heap pressure on long-running ops like `sit log` and `sit fsck` — not measurable in the bench numbers above, but a real reduction in process RSS over thousands of read_object calls.

## What remains (next perf releases)

In rough priority for measurable user-visible wins:

- **P-03** — batch `copy_objects` into a single transaction. Should produce a real clone speedup. The current per-object insert path goes via N WAL-synced INSERTs; a single BEGIN/COMMIT around the loop should collapse most of the syscall overhead.
- **P-06 + P-15** — smarter decompression sizing (read the framing length prefix to size exactly, instead of `blen * 16` + retry); route the LCS DP table through `fl_alloc` so diff-heavy commands don't permanently reserve bump memory.
- **P-04** — `walk_reachable_from_commit` decompresses every commit/tree just to read 1-2 header lines. Cache or denormalize the parent/tree hashes into the patra schema.
- **P-10 + P-18** — hashmap-backed `tree_find` and `three_way_path_set` dedup (currently O(N²) linear scans).
- **P-11** — `sit add` index upsert without full rewrite (needs patra UPSERT).
- **P-17** — buffered stdout to coalesce per-line `write(2)` syscalls.

The patra-cache refactor was the prerequisite for several of these (anything that wants to thread a single open handle through a hot loop). With it landed, the rest of the perf arc unblocks.
