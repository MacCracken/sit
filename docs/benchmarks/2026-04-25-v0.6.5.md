## Where the win is

**Clone -15%.** `copy_objects` previously did three SQL round-trips per object (outer `db_object_has`, inner `db_object_has` from `db_object_insert_raw`, and `patra_insert_row`), each WAL-synced. v0.6.5 drops the outer redundant has-check and wraps the loop in a single `patra_begin` / `patra_commit` — collapses ~300 fsyncs into 1 on the 100-commit / 100-file fixture. Clone goes from 16.13x git to 13.82x git.

The bigger wins still wait on patra-side work — `WAL group commit / batched fsync` and `INSERT OR IGNORE` / `UPSERT` are filed on patra's roadmap (entries cite this consumer + this snapshot). When those land, a follow-on sit release can drop the redundant inner `db_object_has` and skip the manual transaction wrapping; expected combined improvement is another 30-50% on top of v0.6.5's gain, taking clone from 13.82x to ~5-7x git.

## What didn't move

`status-100files` and `add-*`: same story as v0.6.4 — bottlenecks are sigil SHA-256 throughput (filed on sigil's roadmap) and zlib compress/decompress throughput (filed on sankoch's roadmap). Sit-side fixes can't move them; waiting on dep updates.

`log-100commits`, `commit`, `diff-edit`, `fetch-1commit`: not on the P-03 path. Within noise vs v0.6.4 as expected.

## Counting fix

Side-effect of dropping the outer has-check: `db_object_insert_raw` now returns `1` when the object was already present (vs. `0` for "actually inserted"; negative for error). Caller `copy_objects` increments the `copied` counter only on `== 0`. Without this fix, `sit push` on a clone would have reported all reachable objects as "new" instead of just the ones genuinely added on the local side. Caught by the wire-protocol smoke test (cloned 30-commit repo → added 1 commit → push reported 93 instead of 3 new objects). Fixed in the same change.

## Architecture note

`patra_begin` / `patra_commit` are stdlib primitives that have shipped in patra since at least 1.6.0 but were dead code in every sit build prior to v0.6.5 (visible in the build's "dead:" output). After P-03, `patra_commit` is reachable; `patra_begin` is reachable. `patra_rollback` is still dead — sit's copy_objects uses an "always commit" pattern because individual `db_object_insert_raw` failures don't compose cleanly with patra's transaction model in this stdlib version (no per-row error info; rollback would discard valid inserts). Acceptable for now since sit's wire protocol re-fetches missing objects on the next attempt; a future patra release that exposes per-row insert results would unlock proper rollback behavior here.

## Next ship target

Per [the roadmap](../development/roadmap.md), the remaining v0.6.x items in priority order:

1. **P-04** — `walk_reachable_from_commit` decompresses every commit/tree just to read 1-2 header lines. Cache decompressed objects during the walk (sit-side, no dep dependency). Targets clone (the walk is the second-largest component now that copy_objects is batched) and fetch.
2. **P-06 + P-15** — smarter decompression sizing (read the framing length prefix instead of `blen * 16` + retry); LCS DP table through `fl_alloc`. Targets diff and clone.
3. **P-10 + P-18** — hashmap-backed `tree_find` + `three_way_path_set` dedup. Targets status / diff / merge.
4. **P-17** — buffered stdout. Cosmetic on small commands; ~3000 syscalls collapse for a 500-line diff.

When patra ships group commit + UPSERT, revisit P-03 to drop the manual transaction wrap and the inner has-check; expected combined clone improvement is ~3x further (final clone target: ~70 ms, ~5x git).
