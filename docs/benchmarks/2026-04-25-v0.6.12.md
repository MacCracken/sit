## Where the wins came from — sigil 2.9.3's SHA-NI cascade

The per-primitive deltas tell the story:

| Bench | v0.6.11 | v0.6.12 | factor |
|---|---:|---:|---:|
| `sha256-64B` | 10 µs | **802 ns** | **12.5×** |
| `sha256-1024B` | 87 µs | **3 µs** | **29×** |
| `sha256-65536B` | 5.153 ms | **161 µs** | **32×** |
| `zlib-compress-1024B` | 136 µs | 131 µs | 1.04× (within noise) |
| `zlib-compress-65536B` | 1.236 ms | 1.158 ms | 1.07× (modest) |
| `zlib-decompress-1024B` | 32 µs | 36 µs | within noise |
| `zlib-decompress-65536B` | 330 µs | 346 µs | within noise |

**Sigil**: SHA-256 throughput went from ~12 MB/s software-only to ~**400 MB/s** on 64 KB inputs (sigil 2.9.3 picked up the SHA-NI hardware path). The 32× factor on 65536B exactly maps to the gap I'd projected when filing the throughput investigation on sigil's roadmap.

**Sankoch**: 2.1.0 ships incremental DEFLATE wins (the down-payments captured in sankoch's roadmap — pre-reversed dynamic Huffman codes, others). The standard `zlib_compress` / `zlib_decompress` paths the bench measures move only modestly (~5-7% on compress, within noise on decompress). Bigger sankoch wins are queued for 2.x point releases that target the LZ77 match-finder and ring-buffer rewrite.

### How sigil's SHA-256 win cascades into `sit add`

`add-64KB` budget (post-v0.6.12, ~9.6 ms total):
- Sigil SHA-256(64KB framed): ~161 µs (was 5.2 ms — saved ~5 ms)
- Sankoch zlib_compress(64KB): ~1.2 ms (saved ~80 µs from sankoch 2.1)
- Patra insert + index update: small
- File I/O + setup: ~0.5 ms
- Sigil portion of the 7.4 ms savings: **~99% of the win**

`add-1MB` budget (post-v0.6.12, ~112 ms total):
- Sigil SHA-256(1MB framed): ~3 ms (was ~80 ms — saved ~77 ms)
- Sankoch zlib_compress(1MB): ~140 ms (saved ~10 ms from sankoch 2.1)
- Other: ~5 ms
- Sigil portion of the 121 ms savings: **~85%**

The remaining `add-1MB` budget is now sankoch zlib_compress at ~140 ms. That's the next ceiling — and it's exactly what sankoch's roadmap is targeting for the libdeflate-class throughput investigation.

## Why other rows didn't move

- **`status-100files`** (−8%, modest): sigil SHA-256 over 100 tiny files (~17 bytes each) was ~1 ms total pre-v0.6.12, now ~80 µs. Saved ~900 µs of the 7 ms total. Remaining cost is 100× file open+read (~3-4 ms) and per-file path validation + index lookup. Status is now I/O-bound, not crypto-bound.
- **`log-100commits`** (within noise): walks 100 commits, per-commit cost is `read_object` (patra read + sankoch decompress + commit-body parse). Decompress on small commit bodies wasn't moved by sankoch 2.1; patra cache (v0.6.4) already removed the per-call open cost; sigil isn't on this path. Next mover is sankoch decompress throughput on small inputs OR patra row-batch readout.
- **`clone-100commits`** (within noise): per-object insert + decompress + materialize file_write. v0.6.5 P-03 batched the inserts; v0.6.7 P-04 cached the walk decompresses. Remaining: per-object decompress during materialize (sankoch on small inputs) + per-file file_write (kernel-bound). Sankoch 2.1 didn't move small-input decompress; patra's `INSERT OR IGNORE` programmatic flag (filed but not yet shipped) would let us drop one more SELECT per object.
- **`diff-edit`** (within noise): zlib_decompress + LCS. Sankoch 2.1 didn't move decompress at the 1KB-blob scale; LCS algorithm is sit-side and audit P-14 (Myers' diff) is deferred to v0.8.0.

## Where the next wins live

With sigil's SHA-256 throughput largely solved, the remaining "sit slower than git" rows are now:

| sit op currently slow | dominant cost | who ships the next win |
|---|---|---|
| `clone` 11.44× | patra per-insert overhead (~150µs/object × 300 = ~45ms of the 174ms) + sankoch decompress per object | patra programmatic `INSERT OR IGNORE` (filed); sankoch 2.x small-input decompress |
| `log` 5.78× | sankoch decompress per commit (~30ms total) + per-commit parse | sankoch small-input decompress |
| `add-1MB` 6.50× | sankoch `zlib_compress(1MB)` ~140ms | sankoch's 2.x match-finder / ring-buffer / SIMD work |
| `diff` 4.39× | LCS algorithm + sankoch decompress | sit-side P-14 Myers' diff (deferred to v0.8.0); sankoch decompress |
| `add-64KB` 2.55× | sankoch `zlib_compress(64KB)` ~1.2ms + small constant | sankoch 2.x DEFLATE work |

The picture is now: the dep ecosystem has paid down ~half the historical sit-vs-git gap; remaining wall-clock is split between patra's insert path and sankoch's compress/decompress on the relevant input sizes. Both are filed on those repos' roadmaps.

## Setup honesty

- Per-primitive numbers are single-iteration `cyrius bench` measurements; `min == avg == max` because the bench loop runs many internal iterations per measurement.
- The git-vs-sit run is `RUNS_LIGHT=20 RUNS_HEAVY=10`. Run-to-run variance across snapshots is ±5-10% on small ops (e.g. `init`, `commit`, `add-1KB` are noise-bound at this scale on this hardware). Headline deltas (sigil 30× factor, `add-1MB` halved) are well outside any noise band.
- No source changes shipped in v0.6.12 — pure dep bumps. Every win on this page is upstream work picked up cleanly via `cyrius.cyml` pin updates.
