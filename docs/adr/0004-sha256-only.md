# 0004 — sit is SHA-256 only; no SHA-1 interop

**Status**: Accepted
**Date**: 2026-04-24

## Context

Git has two object-ID formats: SHA-1 (legacy, default on most installations as of this writing) and SHA-256 (experimental → supported but opt-in). Most git repos in the wild use SHA-1.

SHA-1 is cryptographically broken. The [Shattered](https://shattered.io/) attack (2017) demonstrated the first SHA-1 collision; the cost of collision has dropped in the years since. Git has added a [hardened SHA-1](https://github.com/cr-marcstevens/sha1collisiondetection) check that detects known-collision preimages and rejects them — mitigation, not a fix.

sit uses SHA-256 exclusively. Its object framing is byte-compatible with git's SHA-256 format (so the same `"blob 12\0hello, sit!\n"` bytes hash to the same 64-char hex as git in SHA-256 mode). SHA-1 is not implemented in sigil, not accepted in sit's object store, and not reachable through any code path.

The audit surfaced this as worth pinning in an ADR. The risk isn't "sit accidentally accepts SHA-1 today" — it doesn't. The risk is **downgrade attacks**: if sit ever grows a "use legacy git repos as remotes" feature, a motivated attacker could demand SHA-1 on the wire, and a well-meaning interop author could add SHA-1 code paths that reopen the exposure.

## Decision

**sit's object model is SHA-256 only, forever.** sigil will not implement SHA-1 for sit's use. sit's wire protocol, object store, and ref format will never accept a SHA-1 object, even for interop with legacy git repositories.

If sit ever needs to ingest content from a SHA-1 git repo, the pipeline is: re-hash the content under SHA-256 at the boundary, discard the SHA-1 identity, and treat the result as a fresh sit object. The SHA-1 hash never enters sit's trust boundary.

## Consequences

- **Positive**
  - Structurally immune to SHA-1 collision attacks. No preimage mitigation needed.
  - No dual-hash bookkeeping, no compatibility table, no "which ID am I looking at?" reasoning.
  - Simpler sigil surface — one hash, one code path.
  - Consistent with sit's first-party thesis ([ADR 0001](0001-no-ffi-first-party-only.md)): we chose SHA-256 because it's the right long-term choice, not because legacy compatibility forced it.
- **Negative**
  - sit cannot directly clone or fetch from a SHA-1 git repo. Users of legacy repos need to convert, which is a one-time migration step.
  - sit's object IDs don't match most public git repos' IDs. If a user cross-references by hash between git and sit, they won't line up unless the git repo is SHA-256-configured.
  - Re-hashing at the boundary means historical commit metadata in an imported repo has a different commit hash than it did upstream — any external links to the original git SHA-1 break.
- **Neutral**
  - A future `sit import git-sha1 <repo>` command (if it ever exists) is explicitly a content migration, not interop. It re-hashes and emits a fresh sit repo.

## Alternatives considered

- **Dual-hash support with hardened SHA-1.** Git's current approach. Rejected: doubles the trust boundary, keeps a weakening algorithm on the inside of the trust boundary, and every sigil CVE in the SHA-1 path is a sit CVE.
- **SHA-256-only but accept SHA-1-hashed pack files for read-only operations.** A halfway measure. Rejected: any code path that reads a SHA-1 hash is a foothold for a downgrade attack.
- **Defer the decision until an interop need arises.** Rejected: deciding now is free and prevents a future rushed compromise.

## Consequences for wire protocol (v0.6.x+)

HTTP / SSH transports (v0.6.0+ per roadmap) will never advertise a SHA-1 capability. Any wire-level content-addressing uses 64-hex SHA-256 strings. The wire format is fixed at design time.
